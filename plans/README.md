## Findings considered and rejected (so they aren't re-audited)

- **LM Studio "SSRF" / block private IP ranges** — by-design: the user configures
  their own local/LAN LLM endpoint; blocking loopback/private ranges would break
  the core feature. Not a vulnerability.
- **Auto-discard of failed recordings** (`RecordingCoordinator.swift`, the
  transcription-failure path) — only deletes meetings with zero segments AND zero
  notes; discarding a genuinely empty failed capture is defensible.
- **`startTimer` double-invalidate** — `stop()` already invalidates the timer and
  double-start is guarded; no real bug.
- **Calendar-data prompt injection into the local LLM** — worst case is garbage
  notes for one meeting against the user's own local model; very low leverage.
  (Optional future hardening: quote/escape title+attendees in prompts.)
- **Linter/formatter config** — the code is already clean and consistent;
  low leverage. Add `.swiftlint.yml` later if CI is introduced.
- **URL string-concatenation in `LMStudioClient.makeURL`** — minor; `URLComponents`
  would be tidier but there's no real defect (user-controlled local URL).
