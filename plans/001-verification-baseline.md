## Why this matters

Oatmeal has **zero automated tests** — no test target, no `XCTest`/`Testing`
imports anywhere in 13.3K lines. Its git history is largely a sequence of
crash fixes (multiple SwiftData deleted-object traps). There is no one-command
way to know the codebase still works, which makes every refactor in the other
plans riskier than it needs to be. This plan creates the test target and
covers the highest-value **pure** logic first (date parsing, vector math, the
fragile JSON-extraction from LLM responses). It is the prerequisite for the
refactor plans (007, 010).

## Steps

### Step 1: Add a test target to `project.yml`

Add under `targets:` (sibling of `Oatmeal` and `OatmealMCP`):

```yaml
  OatmealTests:
    type: bundle.unit-test
    platform: macOS
    sources:
      - path: OatmealTests
    dependencies:
      - target: Oatmeal
    settings:
      base:
        PRODUCT_BUNDLE_IDENTIFIER: com.oatmeal.OatmealTests
        SWIFT_VERSION: "5.0"
        MACOSX_DEPLOYMENT_TARGET: "14.0"
        GENERATE_INFOPLIST_FILE: YES
```

Create an empty `OatmealTests/` dir with at least one file (Step 2) before
regenerating, or XcodeGen will warn about an empty source path.

**Verify**: `xcodegen generate` → exit 0. Then confirm the scheme sees the
tests: `xcodebuild -project Oatmeal.xcodeproj -list` → output includes
`OatmealTests` under Targets.

### Step 2: Write `TaskDatesTests.swift` and `EmbeddingsTests.swift` (public API only)

`TaskDates` and `Embeddings` are `enum`s with `static` members that are
**internal** (no access modifier) — a `@testable import Oatmeal` reaches them
without any production change.

`OatmealTests/EmbeddingsTests.swift` — cover `cosine`:
- identical non-zero vectors → `1.0` (within 1e-5);
- orthogonal vectors `[1,0]`,`[0,1]` → `0.0`;
- length mismatch `[1,2]`,`[1]` → `-1`;
- empty inputs → `-1`;
- a zero vector → `-1` (zero-norm guard).
Do NOT test `vector(for:)` — it depends on `NLEmbedding` model availability,
which may be absent in CI; note that in a comment.

`OatmealTests/TaskDatesTests.swift` — cover `bucket(for:)`. `ActionItem` is a
SwiftData `@Model`; constructing one needs an in-memory `ModelContainer`. Use:
```swift
let container = try ModelContainer(
    for: Meeting.self, ActionItem.self, /* ...other @Model types as required */
    configurations: ModelConfiguration(isStoredInMemoryOnly: true))
```
If the model graph is large, prefer testing `bucket(for:)` by constructing the
minimal `ActionItem` and setting `dueDate`/`isDone`/`snoozedUntil`. Cover:
done → `.done`; future `snoozedUntil` → `.snoozed`; nil `dueDate` → `.noDate`;
due yesterday → `.overdue`; due later today → `.today`; due in 3 days →
`.thisWeek`; due in 30 days → `.later`. Compute relative dates from
`Date.now` inside the test.

**STOP** if building the in-memory `ModelContainer` requires types that pull in
non-testable dependencies — if so, test only `TaskDates.parse` + `Embeddings`
in this step and report the `ActionItem` blocker.

**Verify**: `xcodebuild test ...` (command above) → `** TEST SUCCEEDED **`, and
the log lists the new test cases.

### Step 3: Test the private JSON extractors with the narrowest possible change

`extractJSON(from:)` and `extractArray(from:)` are `private`. Do NOT make them
`public`. Change `private func` to `func` (internal) on exactly these two
methods so `@testable import` reaches them:
- `Oatmeal/Summary/SummarizationService.swift:134` — `private func extractJSON` → `func extractJSON`
- `Oatmeal/Summary/ActionItemExtractor.swift:56` — `private func extractArray` → `func extractArray`

Leave a one-line comment on each: `// internal (not private) for unit tests`.

`OatmealTests/JSONExtractionTests.swift` — these are the fragile parsers that
break when the local LLM wraps JSON in prose/markdown. Cover:
- clean object `{"a":1}` → returns it unchanged;
- object with leading prose `Here you go: {"a":1}` → returns `{"a":1}`;
- object inside ```` ```json ... ``` ```` fences → returns the object;
- no braces → returns `nil`;
- `}` before `{` (`} text {`) → first `{`..last `}` still spans; assert the
  documented behavior (it returns the slice; record what it actually does);
- array variants for `extractArray` (`[...]`, prose-wrapped, none → nil).

These tests document current behavior (characterization) — if a case looks
buggy, capture the ACTUAL output in the assertion and note it in
`Maintenance notes`; do not "fix" the parser here (that's plan 007).

**Verify**: `xcodebuild test ...` → `** TEST SUCCEEDED **`; new tests run.
